### 2. cert-manager — Breaking change in v1.18.0 (SKIP)

| Chart | File | Current | Latest Available | Reason |
|-------|------|---------|------------------|--------|
| cert-manager | argocd/platform/cert-manager.yaml | v1.16.2 | v1.20.2 | **SKIPPED — Breaking change in v1.18.0** |

**Details:** cert-manager v1.18.0 introduced two breaking changes:
1. **`Certificate.Spec.PrivateKey.RotationPolicy` default changed to `Always`** — previously
   certificates would not rotate their private key by default; after this change they will. This
   can cause unexpected certificate re-issuance and short-lived interruptions.
2. **`Certificate.Spec.RevisionHistoryLimit` default changed to `1`** — previously unlimited
   history; this may affect operations relying on certificate history.

Additionally, v1.19.0 had a known issue with unexpected certificate renewal (fixed in v1.19.1).

The jump spans four minor versions (v1.16.2 → v1.17.x → v1.18.x → v1.19.x → v1.20.x). Each should
be reviewed independently.

**Action required:**
- Review `Certificate` resources in the cluster for explicit `rotationPolicy` settings; add
  `rotationPolicy: Never` if the default rotation-on-renewal is not desired.
- Plan a maintenance window for the upgrade path.
- Full release notes: https://github.com/cert-manager/cert-manager/releases

> upgrade caused complete data loss (PR #10, chart transition that caused PG 16→17 jump).

The task specification incorrectly assumed bitnami/postgresql chart 16.x corresponds to
PostgreSQL 16.x. The actual mapping is offset:

| Bitnami Chart Version | PostgreSQL appVersion | PG Major |
|-----------------------|----------------------|----------|
| 15.0.0 | 16.2.0 | **16** |
| 15.5.22 | 16.4.0 | **16** |
| 16.2.4 (current) | 17.2.0 | **17** |
| 16.7.27 | 17.6.0 | **17** |
| 17.1.0 (latest) | 17.6.0 | **17** |

**Both chart 16.2.4 and chart 17.1.0 use PostgreSQL 17.x.** The PostgreSQL *major* version does
not change with this upgrade (17.2 → 17.6). `initdb` is only triggered on PostgreSQL major version
changes, so **no data loss risk** from this specific upgrade per Bitnami's documented behavior.

The previous incident (PR #10) involved a chart upgrade that changed the PostgreSQL major version
(which appears to have been chart 15.x → 16.x, transitioning PG 16 → PG 17). We are already past
that transition. This update (16.x → 17.x) stays within PG 17.

**Nevertheless:** Human review is recommended before merging this specific change. Verify PG version
in the running pod with `psql --version` before and after applying. Monitor the frolf-bot Postgres
pod startup logs for any unexpected `initdb` activity.
